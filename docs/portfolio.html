1,300만 건의 **H&M Kaggle Dataset**을 기반으로 구축된 대규모 트래픽 지향 이커머스 시스템입니다. 단순 API 개발을 넘어 **CQRS 패턴을 통한 검색 성능 최적화**, **Redis 분산 세션 관리**, **마르코프 체인 추천 알고리즘** 적용 등 백엔드 아키텍처의 견고함과 데이터 기반 가치 창출에 집중했습니다.

핵심 성과 (Key Results)

Performance 검색 성능 8.02배 향상

다중 조인으로 인한 DB 부하를 해결하기 위해 **CQRS 패턴 및 역정규화 테이블**을 구축했습니다. 평균 Latency를 101ms에서 12.6ms로 단축했습니다.

Scalability API 응답 35배 개선

추천 시스템 연산 부하를 줄이기 위해 **Redis 캐싱 전략**을 도입했습니다. 반복 요청에 대해 DB I/O를 제거하여 응답 속도를 210ms에서 6ms로 개선했습니다.

Stability 인증 시스템 결함 0%

RDB 기반 인증의 Race Condition 문제를 **Redis In-Memory 저장소**로 이전하여 해결했습니다. 고부하 테스트 중 에러율을 0.64%에서 0%로 안정화했습니다.

AI/ML 추천 정확도 10배 달성

자체 개발한 **마르코프 체인 모델**이 기존 인기 상품(Popularity) 추천 모델 대비 **MAP@20 기준 10배 이상의 정확도**를 기록했습니다.

Ops 고가용성 인프라 구축

**Docker Swarm**을 이용한 수평 확장으로 단일 장애 지점(SPOF)을 제거하고, **GitHub Actions**로 무중단 CI/CD 환경을 구축했습니다.

Database Design (ERD)

  • 도메인 중심 설계: Auth, User, Product, Order 도메인을 명확히 분리하여 MSA 전환이 용이한 구조를 지향했습니다.
  • CQRS 패턴 구현: `ProductSearch` 테이블을 별도로 두어, 정규화된 데이터의 무결성을 지키면서도 검색 시에는 다중 조인 없는 고성능 조회를 가능케 했습니다.
  • 마르코프 전이 확률 저장: `CoPurchase` 테이블을 통해 추천 엔진이 계산한 상품 간 연관 점수를 실시간 API에 제공합니다.

주요 기술 과제 및 해결 경험

1. 검색 성능 최적화 (CQRS 및 역정규화)

  • Challenge: 5개 이상의 테이블 조인으로 인한 검색 API의 높은 Latency(101ms) 및 DB 부하 문제.
  • Action:
    • CQRS 패턴 도입: 단일 DB 내 읽기 전용 역정규화 테이블을 구축하여 다중 조인 연산을 근본적으로 제거.
    • 데이터 정합성 보장: 도메인 이벤트 기반의 비동기 처리를 통해 쓰기 모델과 읽기 모델 간의 최종 일관성 확보.
    • 인덱스 최적화: 역정규화된 테이블에 복합 인덱스를 적용하여 다양한 검색 조건에서도 O(log N) 성능 보장.
    • Pros조회 성능 극대화, 복잡한 JOIN 제거로 인한 DB 부하 감소
    Cons데이터 중복 저장 및 모델 간 일시적 데이터 불일치(Eventual Consistency) 발생 가능성
    Decision이커머스 검색 특성상 실시간성보다 응답 속도가 UX에 미치는 영향이 크다고 판단하여 도입 결정
    Result: 평균 응답 속도 101ms → 12.6ms (8.02배 향상) 및 DB CPU 점유율 대폭 감소.

2. API 응답 속도 극대화 (Caching)

  • Challenge: 추천 시스템 등 연산 집약적이거나 반복적인 DB 조회가 필요한 API의 Latency 최소화.
  • Action: 변경 빈도가 낮은 데이터에 Redis 기반 캐싱 전략을 적용. 반복 요청은 DB를 거치지 않고 메모리에서 즉시 결과를 반환하도록 구현.
    • ProsDB I/O를 메모리 접근으로 대체하여 응답 속도 비약적 향상 및 인프라 비용 절감
    Cons캐시 데이터 갱신(Invalidation) 정책 설정의 복잡성 및 메모리 자원 추가 소모
    Decision조회 빈도가 높고 데이터의 실시간 갱신이 상대적으로 덜 민감한 추천 데이터의 특성을 고려
    Result: 평균 응답 속도 210ms → 6ms (35배 향상) 및 DB I/O 부하 대폭 절감.

3. 인증 시스템 안정성 및 확장성 확보

  • Challenge: RDB 기반 인증 방식이 고부하 상황에서 Race Condition 발생(0.64% 에러율) 및 I/O 병목 유발.
  • Action:
    • In-Memory 저장소 활용: 토큰 저장소를 Redis로 전환하여 디스크 I/O 병목 해결.
    • Stateless 아키텍처: JWT와 Redis를 결합하여 서버 확장(Scale-out) 시에도 정합성 유지.
    • Pros동시성 제어 안정성 확보 및 세션 불일치 문제 근본적 해결
    ConsRedis 인프라 관리 비용 및 추가적인 네트워크 홉 발생
    Decision고부하 상황에서의 무결성과 수평 확장성 확보를 위해 인메모리 기반 세션 관리 방식 선택
    Result: 에러율 0% (결함 제로) 달성 및 인증 API 응답 속도 3.2배 향상.

4. 데이터 기반 추천 시스템 개발

  • Challenge: 개인화된 상품 추천을 통해 구매 전환율을 높여야 하는 비즈니스 과제.
  • Action: 사용자 구매 이력을 기반으로 다음 구매 상품을 예측하는 Markov Chain 모델 개발 및 MAP, NDCG 지표를 통한 검증.
    • Pros단순 인기순 대비 높은 개인화 정확도(10배 이상) 및 구매 전환율 기여
    Cons모델 연산량 증가 및 학습 데이터 관리를 위한 추가 파이프라인 필요
    Decision서비스 경쟁력 확보를 위해 고정밀 모델을 선택하고, 배치를 통해 연산 부하 분산

인프라 및 개발 문화

운영 및 품질 관리

  • 고가용성 아키텍처 (Scale-Out): Docker Swarm을 이용한 수평 확장으로 단일 장애 지점(SPOF)을 제거하고 트래픽 급증에 유연하게 대응합니다.
  • 인프라 자동화 (CI/CD): GitHub Actions와 Docker를 활용하여 테스트, 빌드, 배포 파이프라인을 자동화하고 재현 가능한 환경을 구축했습니다.
  • 모니터링: Prometheus, Grafana 기반의 대시보드를 구축하여 시스템 상태를 실시간으로 추적하고 장애 발생 시 빠른 원인 분석이 가능합니다.
  • 코드 품질: **TDD(Test Driven Development)** 방식의 테스트 코드 작성을 지향하고, 상세한 API 문서와 UML 다이어그램을 통해 협업 효율을 높였습니다.

향후 과제 및 기술 비전

  • MSA 전환 준비: 현재 모놀리식 아키텍처의 한계를 극복하기 위해 **CDC(Change Data Capture) 또는 이벤트 기반 아키텍처**를 도입하여 서비스 간 결합도를 낮출 계획입니다.
  • AI 기술 접목: 추천 모델 고도화 및 **LLM(Large Language Model)**을 활용한 지능형 상품 검색 등 사용자 경험을 혁신하는 기술을 지속 연구하고 있습니다.
