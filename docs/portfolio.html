단순히 '돌아가는' 기능을 만드는 것에 만족하지 않았습니다. 1,300만 건의 대규모 데이터셋을 마주하며, **"어떻게 하면 수많은 트래픽 속에서도 0.01초의 지연 없이 최적의 상품을 제안할 수 있을까?"**라는 질문에 답하기 위해 고민했습니다. CQRS를 통한 검색 최적화부터 Redis 분산 세션, 마르코프 체인 기반의 추천 엔진까지, 백엔드 엔지니어로서 성능과 안정성 사이의 최적점을 찾아가는 과정을 담았습니다.

수치로 증명한 엔지니어링의 가치

Performance "검색 성능 8.02배의 비약적 향상"

다중 조인의 늪에 빠진 DB 부하를 해결하기 위해 **CQRS 패턴**을 도입했습니다. 읽기 전용 테이블을 분리하여 평균 응답 속도를 101ms에서 12.6ms로 단축시키는 과정을 거쳤습니다.

Scalability "캐싱 전략이 만든 35배의 응답 혁신"

반복되는 추천 연산의 DB I/O를 제거하기 위해 **Redis 기반 캐싱**을 설계했습니다. 잦은 호출에도 시스템이 지치지 않도록 구조를 개선하여 응답 속도를 210ms에서 6ms로 최적화했습니다.

Stability "결함률 0%를 향한 무결성 설계"

고부하 상황에서 발생하던 인증 Race Condition을 해결하고자 세션 저장소를 **Redis**로 이전했습니다. 고강도 부하 테스트 결과, 에러율을 0.64%에서 완벽한 0%로 안정화했습니다.

AI/ML "데이터로 증명한 10배의 추천 정밀도"

단순 인기순 추천을 넘어, 사용자 흐름을 읽는 **마르코프 체인 모델**을 직접 구현했습니다. 기존 모델 대비 MAP@20 기준 10배 이상의 정확도를 기록하며 데이터의 가치를 증명했습니다.

Ops "멈추지 않는 서비스를 위한 인프라"

**Docker Swarm**을 이용한 수평 확장과 **GitHub Actions** 기반의 무중단 배포 환경을 구축하여, 어떤 상황에서도 서비스가 지속될 수 있는 견고한 운영 체계를 마련했습니다.

기술적 난관과 이를 통한 성장

01. 복잡한 JOIN의 늪에서 탈출하기 (CQRS 도입)

  • 고민의 시작: 서비스 규모가 커지며 검색 한 번에 5개 이상의 테이블을 조인하게 되었고, 응답 시간이 100ms를 넘어서며 DB CPU 점유율이 치솟았습니다.
  • 해결의 열쇠: 단순히 하드웨어를 늘리는 대신 **CQRS 패턴**을 선택했습니다. 조인이 필요 없는 '검색 전용 읽기 모델'을 구축하고, 도메인 이벤트를 통해 데이터 정합성을 비동기로 맞추는 구조를 설계했습니다.
    • Pros데이터 조인 연산을 0으로 수렴하게 하여 극단적인 조회 성능 확보
    Cons쓰기 모델과 읽기 모델 간의 최종 일관성(Eventual Consistency)에 대한 관리 필요
    Decision사용자 경험상 상품 정보의 몇 초간의 업데이트 지연보다, 검색 결과가 나오지 않는 지연이 더 치명적이라 판단하여 도입을 결정했습니다.
    결과: 평균 응답 속도는 101ms에서 12.6ms로 8배 이상 단축되었습니다.

02. API 응답 속도 극대화 (Redis Caching 전략)

  • 고민의 시작: 추천 시스템 등 연산 집약적인 로직이 매번 DB를 거치며 전체 API 성능의 병목 지점이 되었습니다. 특히 반복적인 조회 요청은 불필요한 DB I/O를 발생시켰습니다.
  • 해결의 열쇠: 데이터의 변경 빈도가 낮고 조회 빈도가 높은 특성을 활용해 **Redis 기반 캐싱 레이어**를 구축했습니다. Cache-Aside 전략을 통해 반복되는 요청은 메모리에서 즉시 반환하도록 구현했습니다.
    • ProsDB 접근을 최소화하여 응답 속도를 비약적으로 향상시키고 인프라 비용 절감
    Cons캐시 데이터 갱신(Invalidation) 정책의 복잡성 및 메모리 자원 추가 관리 필요
    Decision데이터의 실시간성보다 시스템 전체의 응답성이 핵심인 추천 도메인의 특성을 고려하여 적극적인 캐싱을 채택했습니다.
    결과: 평균 응답 속도가 210ms에서 6ms로 약 35배 빨라졌으며, DB I/O 부하를 90% 이상 제거했습니다.

03. 인증 시스템 안정성 및 확장성 확보 (Stateless 아키텍처)

  • 고민의 시작: 기존 RDB 기반 인증 방식이 고부하 상황에서 Race Condition을 유발하며 0.64%의 에러율을 기록했습니다. 또한, 서버 확장 시 세션 동기화 문제라는 확장성의 한계에 부딪혔습니다.
  • 해결의 열쇠: 토큰 저장소를 **Redis In-Memory**로 이전하여 I/O 병목을 해결하고, JWT를 결합한 **Stateless 아키텍처**를 완성했습니다. 이를 통해 모든 서버 인스턴스가 동일한 인증 상태를 공유할 수 있게 되었습니다.
    • Pros동시성 제어의 안정성 확보 및 완벽한 수평 확장(Scale-out) 환경 구축
    ConsRedis 인프라 관리 리소스 및 추가적인 네트워크 홉 발생
    Decision대규모 트래픽 지향이라는 목표를 달성하기 위해, 관리가 용이한 RDB보다 성능과 확장성에 유리한 인메모리 방식이 필수적이라 판단했습니다.
    결과: 고강도 테스트 중에도 에러율 0% (결함 제로)를 달성했으며, 인증 API 응답 속도 또한 3.2배 향상되었습니다.

04. 사용자 맞춤 추천을 위한 데이터 엔지니어링

  • 고민의 시작: "사용자가 다음에 구매할 물건을 미리 알 수 없을까?"라는 비즈니스적 요구에 부딪혔습니다. 단순 인기순 추천은 개인화된 경험을 제공하기에 부족했습니다.
  • 해결의 열쇠: 수만 명의 사용자 동선을 분석하여 **마르코프 체인 전이 확률 모델**을 구축했습니다. 이를 통해 사용자의 현재 장바구니 상태와 과거 이력에 기반한 최적의 '다음 구매 상품'을 제안하는 엔진을 개발했습니다.
    • Pros사용자 맞춤형 쇼핑 경험 제공으로 실질적인 구매 전환율 상승 기여
    Cons대량의 로그 데이터를 전처리하고 모델을 학습시키기 위한 인프라 비용 발생
    Decision서비스만의 고유한 경쟁력을 확보하기 위해 고성능 추천 엔진이 필수적이라 판단, 배치를 통해 연산 부하 분산 전략을 함께 세웠습니다.
    결과: 10배 높은 지표 향상은 단순한 수치를 넘어 실질적인 매출 최적화로 이어졌습니다. **MAP@20의 비약적 향상**은 추천 리스트 내에 사용자가 실제 구매할 상품이 포함될 확률을 극대화하여 **사용자의 상품 탐색 비용을 획기적으로 낮췄으며**, **NDCG@20의 개선**은 가장 선호도가 높은 상품을 최상단에 노출함으로써 **구매 결정 여정을 단축**시켰습니다. 이러한 정교한 개인화는 장바구니 전환율(CVR) 상승과 추가 구매 유도를 통한 평균 주문 금액(AOV) 증대로 이어져, **데이터 엔지니어링이 비즈니스 수익으로 직결됨**을 증명했습니다.

개발 그 이상의 가치: 운영과 품질

  • 확장 가능한 구조: Docker Swarm을 통해 트래픽 급증 시 서버를 유연하게 늘리는 구조를 만들었습니다.
  • 자동화된 신뢰: **TDD**를 통해 코드 한 줄에도 근거 있는 자신감을 가졌으며, GitHub Actions로 '누르면 배포되는' 안전한 파이프라인을 완성했습니다.
  • 가시성 확보: Prometheus와 Grafana 대시보드를 구축하여, 장애가 발생하기 전 징후를 먼저 발견할 수 있는 관제 체계를 갖췄습니다.
